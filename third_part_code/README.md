## medLLMbenchmark Preprocessing Notes

The preprocessing dataset used for downstream prediction is generated by:

- `third_part_code/medLLMbenchmark/MIMIC-IV-Ext-Creation.py`
- output file: `third_part_code/medLLMbenchmark/MIMIC-IV-Ext-Triage-Specialty-Diagnosis-Decision-Support.csv`

To make preprocessing reproducible across environments, fill in the MIMIC path variables in `.env.local` (see `.env.example` for the template), then run:

```bash
python3 third_part_code/medLLMbenchmark/MIMIC-IV-Ext-Creation.py
```

The script loads `.env.local` automatically via `python-dotenv`.

Sanity-check targets in upstream scripts:

- Preprocessing script truncates to `2200` rows before writing the decision-support dataset.
- Final prediction postprocessing truncates to `2000` rows.

### `df` vs `df_small` — benchmark slice analysis

The script builds a fully filtered `df` then writes only `df_small = df[:2200]` to CSV. The `2200` buffer is intentional: ~200 rows are expected to be dropped in downstream post-processing, leaving ~2000 usable benchmark rows — the target size for the original paper's evaluation.

Every row in `df` passed the same quality filters applied sequentially in `MIMIC-IV-Ext-Creation.py`:

- triage non-null (line 206: `df.dropna(subset=['acuity'])`)
- tests non-null and length < 3000 chars (lines 207, 265–267)
- HPI length between 50 and 2000 chars (lines 257–263)
- HPI contains no ED-setting mentions — `\bED`, `Emergency room`, `Emergency department`, `ER` (lines 282–295, 338–343)
- diagnosis contains no outcome leakage — `deceased`, `died`, `history of present illness` (lines 347–356)

The cutoff at row 2200 is arbitrary, not quality-driven.

The downstream `additional_postprocess.py` drops exactly 3 hard-coded rows (indices 795, 2176, 1208 — specific LLM output failures) plus a small number of rows with missing `initial_vitals` or empty specialty predictions, then enforces `[:2000]` as a final truncation. The 200-row buffer in `df_small` (2200 rows saved, 2000 used) is a conservative safety margin for these drops, not an expected 10% attrition. The final eval set is exactly 2000 rows.

**Ordering artifact.** No shuffle is applied before the slice. Row order is determined by `triage.csv`'s natural ordering, which MIMIC-IV assigns by `stay_id` — a sequential identifier reflecting real-world chronological admission order (lower `stay_id` = earlier real ED visit). `df[:2200]` therefore skews toward earlier admissions in MIMIC-IV's coverage window. Note: MIMIC-IV's per-patient random date shifts scramble stored date values but do not affect `stay_id` assignment, so the chronological ordering is real even though individual dates are not interpretable.
